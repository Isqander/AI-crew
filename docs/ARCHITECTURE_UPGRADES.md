## Улучшения AI-crew: варианты реализации, оценки, архитектурные последствия

Дата: 2026-02-08

Этот документ — «архитектурная записная книжка» по идеям из `docs/IDEAS.md` + дополнительные требования:
- автоматический деплой на VPS (CI/CD), выдача ссылки на развернутый сайт (авто-домен на `nip.io`)
- подключение CLI‑агентов (Claude Code CLI / Codex CLI) как отдельного исполняющего контура
- интернет-доступ (поиск, скачивание артефактов/картинок)
- отображение/редактирование графа агентов во фронтенде
- анализ прошлых флоу и (контролируемая) эволюция графов и промптов
- управление не только через Web UI, но и через Telegram
- расширение на не‑софтверные сценарии (research/контент/и т.п.)

## Текущее состояние репозитория (важно для решений)

### Что уже есть
- **Оркестрация**: LangGraph (`graphs/dev_team/graph.py`), HITL через `interrupt_before=["clarification", "human_escalation"]`.
- **API сервер**: Aegra (FastAPI, совместим с LangGraph Platform API), конфиг `aegra.json`.
- **Frontend**: React+Vite. Уже есть клиентский метод **SSE streaming** `aegraClient.streamRun()` в `frontend/src/api/aegra.ts`, но основной хук `useTask` использует **polling** (2 секунды).
- **Хранилище**: Postgres + pgvector (в compose).
- **Observability**: Langfuse в compose (опционально).
- **Прод деплой**: есть режим **single-container** (Dockerfile) — Aegra + nginx(front) + Langfuse(v2, опционально) + Postgres внутри контейнера.

### Важные ограничения/наблюдения
- `vendor/aegra/` **не модифицируем** → любые расширения Aegra делаем «снаружи» (gateway/sidecar) или через форк как отдельный пакет (если когда‑то понадобится).
- LLM конфигурация сейчас в `graphs/dev_team/agents/base.py` (дефолты + env).
- Сейчас код «передаётся» между агентами через `state` (со списком `code_files`), а в GitHub пушится только в `git_commit_node` в самом конце. Для больших репо это плохо масштабируется → нужно переходить к **git/workspace‑ориентированному** обмену.

## Шкала оценок (условные единицы)

Чтобы сравнивать идеи «по относительной трудоёмкости», используем шкалу 1–10:
- **1–2**: быстрый инкремент (часы)
- **3–4**: 1–3 дня
- **5–6**: ~неделя
- **7–8**: 2–4 недели
- **9–10**: эпик (1–2 месяца) + высокий риск/неопределённость

Оценки подразумевают, что большую часть делает ИИ‑агент (быстрее человека), но инфраструктура/безопасность всё равно требуют аккуратности.

## 1) Retry логика для LLM вызовов

### Варианты реализации
- **A. Tenacity вокруг `llm.invoke`/`ainvoke`**  
  Классика: экспоненциальная задержка + jitter, лимит попыток, классификация ошибок.
- **B. Использовать встроенные ретраи клиента** (если поддерживается в используемой обвязке)  
  Иногда проще, но обычно меньше контроля (особенно по типам ошибок/телеметрии).
- **C. Ретрай на уровне gateway/proxy** (если LLM прокси свой)  
  Хорошо централизует, но сложнее отладка по конкретным ролям/промптам.

### Рекомендация
**A (Tenacity) + отдельная функция‑обёртка в `base.py`** с:
- лимитами по времени (deadline), не только по попыткам
- ретраем только на retryable‑ошибки (429/5xx/timeouts)
- метриками (сколько ретраев, латентность, причина)

### Сложность
**2/10**

### Архитектурные изменения
Не требуется; затрагивает LLM factory и/или BaseAgent invoke.

## 2) Структурированное логирование (correlation: thread/run/node/agent)

### Варианты реализации
- **A. `structlog` + JSON output**  
  Быстро, удобно для Loki/ELK, легко добавлять поля.
- **B. OpenTelemetry (traces+logs) + exporters**  
  Более «enterprise», но тяжелее внедрить.
- **C. Только Langfuse**  
  Полезно для LLM‑вызовов, но не заменяет системные логи и корреляцию событий графа.

### Рекомендация
**A сейчас**, с постепенным добавлением **B** (если нужна сквозная трасса через runner/deployer).

### Что логировать (минимальный набор)
- `thread_id`, `run_id`, `graph_id`, `node`, `agent`, `qa_iteration_count`
- `event` (например `agent.invoke`, `router.route_after_qa`, `deploy.start`)
- `duration_ms`, `error_type`, `retry_count`

### Сложность
**3/10**

### Арх. изменения
Минимальные: общий middleware/инициализация логгера + контекст на run/thread.

## 3) Streaming в Frontend (SSE)

### Текущее положение
Клиентский метод `aegraClient.streamRun()` уже реализован, но UI всё ещё опирается на polling.

### Варианты реализации UI
- **A. Новый хук `useStreamingTask`**  
  Отдельно от polling‑хука. Проще внедрять и откатывать.
- **B. Гибрид в `useTask`: streaming если возможно, иначе polling**  
  Лучший UX, но сложнее состояние и edge cases.

### Что стримить
Есть несколько уровней (зависит от того, что отдаёт сервер):
- **Values stream**: целиком `state.values` «как есть» (просто обновлять store).
- **Events stream**: события узлов/сообщений/токены (богатый UX, сложнее парсинг).

### Рекомендация
**B**: streaming по умолчанию + fallback на polling.  
Сначала стримить **values**, затем постепенно добавить более тонкие события (node enter/exit, сообщения, прогресс).

### Сложность
**4/10**

### Арх. изменения
Не требуется. Плюс: можно использовать эти же события для «графа на фронте» (подсветка активного узла).

## 4) Вынести конфигурацию LLM в файл

### Зачем
- не хардкодить модели/температуры в `base.py`
- поддержать профили: `dev`, `prod`, `cheap`, `quality`
- централизованно настроить fallback endpoints

### Варианты
- **A. `config/llm.yaml` + env overrides**  
  Файл в репо, секреты остаются в env/secret manager.
- **B. Конфиг в БД (через UI)**  
  Удобно, но нужен контроль версий и аудит.
- **C. GitOps: конфиг меняется только через PR**  
  Максимально безопасно и аудируемо.

### Рекомендация
**A + C**: конфиг в репо (GitOps), секреты — только env/Actions secrets.

### Сложность
**3/10**

### Арх. изменения
Нет (локально). Для UI‑управления в будущем потребуется конфиг‑сервис/аутентификация.

## 5) Code Execution Sandbox (и почему VPS‑деплой не полностью заменяет песочницу)

### Почему «просто деплой на VPS» не равен sandbox
Деплой на VPS хорош для e2e и «реальной проверки», но:
- нужен быстрый **локальный**/изоляционный прогон тестов без доступа к секретам
- сгенерированный код может быть **вредоносным** (случайно или через prompt injection)
- CLI‑агенты и автогенерация требуют безопасного контура исполнения

### Варианты песочницы
- **A. Docker runner (контейнер на той же VPS)**  
  Ограничения: CPU/RAM/timeout, read-only FS, запрет привилегий, сеть по allowlist.
- **B. gVisor/Kata Containers**  
  Лучше изоляция, выше сложность эксплуатации.
- **C. MicroVM (Firecracker)**  
  Максимальная изоляция, но тяжёлый операционный контур.

### Рекомендация (практичный MVP)
**A** как отдельный сервис `runner` (или отдельный хост) с жёсткими политиками:
- job queue (например Redis/BullMQ или Postgres queue)
- лимиты (cgroups), таймауты
- no‑secrets по умолчанию
- артефакты (логи тестов, отчёты) складывать в хранилище/в Postgres

### Сложность
**8/10**

### Арх. изменения
Появляется отдельный «data plane» сервис (runner) и протокол запуска джоб (API).

## 6) Security Agent

### Что делает
Перед созданием PR (и/или перед деплоем) проверяет:
- секреты в коде (`gitleaks`)
- зависимости (OSV / `trivy fs`, `pip-audit`, `npm audit`)
- SAST (`semgrep`, `bandit` для Python, ESLint security для JS/TS)
- базовые практики: CORS, auth, SSRF, опасные команды

### Варианты интеграции
- **A. Новый узел графа `security_agent` между Developer → QA**  
  Ловит проблемы до ревью, но добавляет время.
- **B. Security после QA, но перед `git_commit`**  
  Меньше прерываний, но поздно.
- **C. Параллельно: Security и QA, потом merge результатов**  
  Лучше по времени, сложнее логика графа.

### Рекомендация
**C** (параллельный запуск) как зрелая цель, но начать с **B** (проще).

### Сложность
**5/10**

### Арх. изменения
Нужен доступ к рабочему дереву (см. раздел про git/workspace вместо `code_files`) и среда, где стоят сканеры (лучше в runner).

## 7) Visual Graph Editor (и «править граф на фронте»)

### Основная проблема
LangGraph граф сейчас описан Python‑кодом. «Визуально редактировать» Python‑граф без потери гибкости сложно.

### Варианты
- **A. Read-only визуализация** (быстро и полезно)  
  UI строит граф из декларативного описания (экспорт из Python или отдельный JSON).
- **B. Graph DSL (YAML/JSON) как source-of-truth**  
  Backend на лету компилирует DSL в LangGraph. UI редактирует DSL.  
  Плюсы: редактор реален. Минусы: нужно ограничить выразительность и хорошо валидировать.
- **C. «Редактор генерирует PR»**  
  UI помогает визуально, но итог — PR с изменениями в `graphs/dev_team/graph.py`/DSL.

### Рекомендация
Фаза 1: **A** (визуализация + подсветка текущего узла).  
Фаза 2: **C** (изменения через PR + preview).  
Фаза 3: **B** (если реально нужен no-code редактор).

### Сложность
- A: **4/10**
- C: **7/10**
- B: **9/10**

### Арх. изменения
Для B/C потребуется:
- версионирование графов (graph version id)
- sandbox/preview окружение для проверки графа перед применением
- строгая валидация (чтобы UI не «сломал» прод)

## 8) Self-Improvement Loop (эволюция промптов/графов)

### Что хотим в идеале
- собирать данные о выполненных задачах (промпты, ответы, метрики, результаты)
- считать качество (успешность тестов, количество итераций Dev↔QA, время/стоимость)
- автоматически предлагать улучшения (промптов, роутеров, набора инструментов)
- применять изменения только после проверки/регрессии

### Опасности
- самопроизвольная деградация качества (особенно без строгих тестов)
- «подгонка» под метрики (Goodhart)
- безопасность (prompt injection через внешние источники)

### Рекомендация: только контролируемая эволюция
MVP:
- **observability + dataset**: сохранять run‑трейсы/артефакты
- **offline eval harness**: прогонять набор задач/тестов на новой версии промптов
- изменения попадают в repo как PR, проходят CI

### Где здесь полезен DSPy / EvoAgentX
- **DSPy**: оптимизация промптов/схем рассуждения на датасете с метриками.
- **EvoAgentX**: идеи эволюции графов/агентов — но безопасно только при наличии eval harness и ограниченного пространства поиска.

### Сложность
**9/10**

### Арх. изменения
Нужны:
- хранилище событий/трейсов (минимум Postgres таблицы, лучше event store)
- версия промптов/графов
- CI‑контур для eval (автотесты + «репрезентативные задачи»)

## 9) Деплой на VPS «из коробки» + CI/CD + выдача ссылки на сайт (`nip.io`)

### Цель
После выполнения задачи система должна:
- настроить инфраструктуру (однократно): VPS, reverse proxy, registry pull, секреты
- настроить CI/CD (GitHub Actions)
- при push в `main/master` автоматически обновлять деплой
- опционально: поднимать **preview окружение** и выдавать URL (удобно для QA)

### 9.1 Минимально‑работающий вариант (прод деплой на `main`)

**Паттерн**: GitHub Actions (hosted runner) → build+push (GHCR) → SSH на VPS → `docker compose pull && docker compose up -d`.

Плюсы:
- минимальная сложность
- хорошо работает с текущим Dockerfile (single-container)

Минусы:
- secrets (SSH key) в GitHub Secrets
- деплой «пуш‑based», без сильной изоляции

**Сложность**: **7/10**

### 9.2 Preview окружения (ссылка на `nip.io` на каждый run/PR)

Есть два удобных подхода:
- **A. Traefik/Caddy как reverse proxy + Docker labels**  
  Каждый деплой добавляет сервис с label `Host(`<subdomain>.<ip>.nip.io`)`.
- **B. PR environments через отдельный compose‑project**  
  `COMPOSE_PROJECT_NAME=aicrew-pr-123`, отдельные порты/хосты.

Рекомендация: **Traefik** (потому что динамическая маршрутизация и автоматизация).

**Сложность**: базовый деплой + **3–4** пункта сверху (итого ~**9/10** для «как платформа»).

### 9.3 «Система полностью настраивает CI/CD»

Чтобы агент действительно мог настроить всё сам:
- инструмент GitHub API: создавать workflow files, environments, secrets
- инструмент SSH/Ansible: конфигурировать VPS (docker, firewall, proxy)
- аудит: логировать каждое изменение инфраструктуры (в идеале — PR + changelog)

Рекомендация: добавить отдельную роль/узел **DevOps Agent** (см. ниже).

## 10) CLI‑агент (Claude Code CLI / Codex CLI) как отдельный исполнитель

### Почему нужен отдельный контур
- CLI‑агенты обычно работают в **локальном репо** (git clone), делают изменения, коммитят/открывают PR.
- для больших проектов нельзя «пересылать весь код» между LLM‑агентами через state → нужен shared workspace или git.

### Архитектурный паттерн
**Control Plane** (Aegra/LangGraph) → ставит задачу  
**Execution Plane** (Runner/CLI‑Agent) → клонирует репо, запускает CLI, пушит ветку, открывает PR.

### Варианты реализации
- **A. CLI‑агент как сервис на отдельной VPS**  
  API: `POST /jobs` (repo, branch, instructions, auth), ответ: PR URL + логи.
- **B. CLI‑агент как «режим» runner**  
  Тот же runner, но разные типы job: `test`, `lint`, `claude_code`, `codex`.

Рекомендация: **B** (меньше сервисов), но с возможностью вынести на отдельный хост при росте рисков.

### Сложность
**8/10**

### Риски/безопасность
- нужно жёстко ограничить доступ: allowlist repos/orgs, запрет произвольного shell, no‑secrets по умолчанию
- обязательно «чистые» ephemeral workspaces (удаление после job)

## 11) «Передавать код между агентами» через git/workspace (вместо state `code_files`)

### Почему это нужно
`code_files` в state подходит для демо, но не для реального репо:
- слишком большой объём
- агентам нужно видеть реальный проект, запускать тесты/линтеры
- история изменений важна

### Рекомендация
Перейти к модели:
- у каждого run есть **workspace** (репо на диске runner или volume)
- агенты оперируют не файлами в state, а **git diff/коммитами/ветками**
- state хранит только ссылки: `workspace_id`, `branch`, `commit_sha`, `pr_url`, артефакты тестов

### Сложность
**6/10**

### Арх. изменения
Появляется сервис/модуль «workspace manager» (часть runner).

## 12) Интернет доступ: web search, fetch, скачивание изображений

### Варианты
- **A. Инструменты в LangGraph (tools) + allowlist доменов**  
  search API → fetch URL → сохранить артефакт.
- **B. Отдельный fetch‑proxy сервис**  
  Централизует безопасность (URL allowlist, mime limits, antivirus).

### Рекомендация
Начать с **A**, но:
- ограничить список доменов и размер ответов
- логировать источники (для QA/аудита)
- защититься от prompt injection (помечать «внешний контент» как недоверенный)

### «Скачивать изображения»
Реализуется как tool: `download_image(url) -> local_path + metadata` + опционально OCR/thumbnail.

### Сложность
**4/10**

## 17) Нужно ли уходить «из одного контейнера»?

### Короткий ответ
Для «демо/простого VPS» single-container ок.  
Для **безопасного исполнения кода/CLI** и масштабирования — **да, нужно дробить**.

### Практичная целевая разбивка сервисов (минимум)
- **api/orchestrator**: Aegra + LangGraph runtime (control plane)
- **db**: Postgres (отдельно, с бэкапами)
- **frontend**: отдельно или статикой через nginx/traefik
- **runner**: выполнение тестов/сканов/CLI (execution plane, изолированный)
- **reverse proxy**: traefik/caddy (TLS, host routing, preview env)
- **observability**: langfuse (опционально) + лог-стек (Loki/ELK — опционально)

## 18) Оставаться ли на Aegra или писать своё?

### Когда Aegra достаточно
- нужен совместимый API (threads/runs/stream)
- HITL уже работает
- не нужны сложные multi-tenant/auth/плагины прямо в рантайме

### Когда потребуется свой backend (или форк Aegra)
- сложная аутентификация/авторизация, тенанты, quotas
- полноценный Visual Graph Editor (с динамической компиляцией)
- глубокая интеграция с runner/deployer как частью «платформы»

### Рекомендованная стратегия
Фаза 1: **оставляем Aegra**, добавляем **gateway/service-layer** вокруг него для:
- auth, rate limits, audit
- endpoints: deploy, runner jobs, graph export/import, telegram hooks

Фаза 2: если ограничения Aegra критичны → **свой FastAPI backend**, который:
- держит тот же API контракт или свой
- напрямую запускает LangGraph
- умеет версии графов/конфигов и sandbox preview

## 19) Что полезно взять из других фреймворков/идей

### Agency Swarm / crewAI
Полезны как источник паттернов (roles, tools, workflows), но:
- у нас уже LangGraph даёт более мощную оркестрацию (роутеры, циклы, subgraphs)
- миграция ради миграции обычно не окупается

### DSPy
Реально полезен для **Self-Improvement Loop**:
- формализация метрик качества
- оптимизация промптов/шаблонов под датасет задач

### EvoAgentX
Полезно как «вдохновение» для эволюции графов:
- но только если есть строгий eval harness и ограниченный поиск
- иначе слишком высокий риск деградации и «необъяснимых» изменений
