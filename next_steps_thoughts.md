# AI-crew: Следующие шаги и улучшения

## 1. Визуальная проверка фронтенда (QA с "глазами")

### Текущее состояние
QA-агент сейчас работает **только как code reviewer** — получает код в виде текста и проверяет его
на соответствие требованиям, безопасность, стиль и т.д. Он **не запускает** код, **не видит** как
выглядит интерфейс, **не выполняет** тестов.

### Что нужно для визуальной QA

#### Шаг 1: Среда выполнения на VPS
- Настроить на VPS **изолированную песочницу** (sandbox), где можно безопасно разворачивать
  сгенерированный код.
- Варианты:
  - **Docker-in-Docker**: Генерировать `Dockerfile` как часть выхода Developer-агента,
    запускать контейнер на VPS, дожидаться готовности (health check), получить URL.
  - **Упрощённый вариант**: Для простых фронтенд-проектов (HTML/CSS/JS) — запустить
    `python -m http.server` или `npx serve` во временной директории.
  - **Kubernetes sandbox**: Если VPS уже имеет K8s — создавать временные Pod'ы.

#### Шаг 2: Интеграция скриншотов / Playwright
- Установить **Playwright** (или Puppeteer) на VPS.
- После разворачивания проекта:
  1. Открыть URL в headless-браузере.
  2. Сделать скриншоты ключевых страниц / состояний.
  3. Передать скриншоты мультимодальной LLM (GPT-4o, Claude Vision, Gemini Pro Vision).

#### Шаг 3: Мультимодальный QA-агент
- Обновить QA-промпт для приёма изображений.
- LLM анализирует скриншоты и сравнивает с требованиями:
  - "Есть ли кнопка X?"
  - "Выровнен ли текст?"
  - "Соответствует ли цветовая схема?"
  - "Отображаются ли данные?"
- Это можно добавить как **дополнительный шаг QA** после code review.

#### Шаг 4: E2E тесты (продвинутый вариант)
- Developer-агент может также генерировать Playwright-тесты как часть кода.
- QA-агент запускает их и анализирует результаты.

### Что нужно от пользователя
- **Доступ к VPS** (SSH / API) с правом запускать Docker-контейнеры.
- Либо отдельный **sandbox-сервис** на VPS с API для развёртывания.
- Мультимодальная LLM модель (GPT-4o / Claude / Gemini) — уже может быть доступна через
  существующий proxy.

### Оценка сложности
- Шаг 1 (sandbox): **Средняя** — 2-3 дня разработки
- Шаг 2 (скриншоты): **Лёгкая** — 1 день
- Шаг 3 (мультимодальный QA): **Лёгкая** — 1 день (промпт + передача изображений)
- Шаг 4 (E2E тесты): **Высокая** — 3-5 дней

## 2. GitHub интеграция — текущее состояние и улучшения

### Текущая конфигурация
- **Переменные**: `GITHUB_TOKEN`, `GITHUB_DEFAULT_REPO` (в docker-compose и env.example)
- **Клиент**: PyGithub, инициализируется лениво через `get_github_client()` при наличии токена
- **Кто пушит**: Только нода `git_commit_node` в графе. Она:
  1. Создаёт feature-ветку `ai/task-YYYYMMDD-HHMMSS`
  2. Коммитит все файлы из `code_files` поштучно через GitHub API
  3. Создаёт Pull Request
- **Кто читает**: Существуют LangChain-тулы (`get_file_content`, `list_repository_files`),
  но они **НЕ подключены** ни к одному агенту. Они определены, но не используются.

### Что работает
- Если в UI при создании задачи указать репозиторий (поле `repository`) и задать
  `GITHUB_TOKEN` — код будет запушен в ветку и создан PR.
- Если репозиторий не указан — код выводится в чат как текст (fallback).
- **Новое**: `GITHUB_DEFAULT_REPO` теперь используется как fallback, если пользователь не
  указал репозиторий в задаче.

### Что можно улучшить

#### 2.1 Подключить GitHub-тулы к агентам
Существующие тулы (`get_file_content`, `list_repository_files`) можно дать Architect-
и Developer-агентам, чтобы они могли:
- Прочитать существующий код из репозитория перед разработкой.
- Понять структуру проекта.
- Учитывать существующий стиль кодирования.

**Сложность**: Лёгкая — нужно передать tools в агентов и обновить промпты.

#### 2.2 Чтение кода из репозитория как контекст задачи
- При создании задачи, если указан репозиторий, автоматически прочитать структуру файлов
  и ключевые файлы (README, package.json, requirements.txt, основные модули).
- Передать их в `context` для всех агентов.

**Сложность**: Средняя — нужна логика определения "ключевых" файлов.

#### 2.3 Обновление существующих файлов (не только создание)
Текущий `git_commit_node` уже поддерживает обновление (`repo.update_file`), но Developer
генерирует файлы целиком. Для больших проектов нужен diff-based подход:
- Developer видит текущий файл и генерирует изменения.
- Коммит содержит только модифицированные файлы.

**Сложность**: Высокая — требует изменения в промптах и парсинге.

#### 2.4 GitHub Webhooks для обратной связи
- Получать уведомления о комментариях к PR.
- Автоматически обрабатывать запросы на изменения от ревьюеров.

**Сложность**: Высокая — нужен webhook endpoint и обработка событий.

## 3. Улучшение цикла Dev↔QA (РЕАЛИЗОВАНО)

### Что было
- Бесконечный цикл: QA находит баги → Developer правит → QA снова находит (в т.ч.
  косметические проблемы, которые были изначально в требованиях).

### Что реализовано (текущий PR)
- **Счётчик итераций** `qa_iteration_count` в состоянии графа.
- **Лестница эскалации**:
  1. До 3 итераций → обычный Dev↔QA цикл
  2. После 3 итераций → **Architect escalation** — Архитектор разбирает, какие баги
     критичны, а какие можно принять. Может вернуть в Dev (ещё 3 итерации) или
     одобрить код.
  3. После ещё 3 итераций (Архитектор уже вмешивался) → **Human escalation** — запрос
     к пользователю через HITL-механизм.

### Возможные дополнительные улучшения
- Настраиваемые лимиты итераций через env-переменные.
- Приоритизация багов: critical баги всегда возвращать в Dev, minor — автоматически waive
  после N итераций.
- Тайм-аут по времени (не только по количеству итераций).

## 4. Другие идеи для следующих шагов

### 4.1 Structured Output (JSON mode) для агентов
Заменить текстовый парсинг ответов (regex, keyword search) на structured output через
`with_structured_output()` LangChain. Это уменьшит ошибки парсинга и сделает поведение
более предсказуемым.

### 4.2 Streaming UI
Фронтенд уже имеет polling каждые 2 секунды. Можно перейти на SSE/WebSocket для
real-time обновлений.

### 4.3 Многопроектная поддержка
Возможность работать с несколькими репозиториями одновременно, сохранять контекст проекта
между задачами.

### 4.4 Кэширование контекста репозитория
При повторных задачах к одному репозиторию — не перечитывать файлы, а использовать
кэшированную структуру (обновляя только при изменениях).

### 4.5 Langfuse интеграция для оптимизации промптов
Использовать данные Langfuse для анализа:
- Какие промпты приводят к лучшим результатам.
- Где чаще всего происходит зацикливание.
- Стоимость и время выполнения по агентам.
